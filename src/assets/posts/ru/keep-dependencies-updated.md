Технический долг накапливается незаметно. Одна из самых коварных его форм — устаревшие зависимости.

Недавно я обновил проект с Angular 20 до 21 и DevExtreme с 24.2 до 25.2. Потребовались усилия, но это было по силам. Команды, пропускающие несколько мажорных версий, часто тратят месяцы на догоняние — миграции с Angular 8 до 16 печально известны этим. Разница — в регулярности обновлений.

## Математика отложенных обновлений

Сравните два подхода:

**Подход А**: Обновляться каждые 2 месяца, тратить 1 день каждый раз. Годовая стоимость: 6 дней.

**Подход Б**: Обновляться раз в год, тратить 2–4 недели. Годовая стоимость: 10–20 дней.

Разрыв растёт с каждым пропущенным циклом. Breaking changes накапливаются. Устаревшие API множатся. Документация по миграциям исчезает. Поддержка сообщества для старых версий сходит на нет.

## Почему команды пропускают обновления

- «Работает — не трогай»
- Нет выделенного времени в спринтах
- Страх регрессий
- Отсутствие тестового покрытия

Каждая причина понятна. Каждую потом дороже устранять.

## Практическая стратегия обновлений

### 1. Автоматизируйте очевидное

Используйте Dependabot или Renovate. Настройте автослияние патч-обновлений при проходящем CI. Для минорных версий создавайте PR автоматически, но требуйте ручной ревью.

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: 'npm'
    directory: '/'
    schedule:
      interval: 'weekly'
    groups:
      minor-and-patch:
        patterns:
          - '*'
        update-types:
          - 'minor'
          - 'patch'
```

### 2. Читайте CHANGELOG перед обновлением

Пять минут на чтение CHANGELOG экономят часы отладки. Ищите:

- Breaking changes (обычно помечены ⚠️ или BREAKING)
- Устаревшие API, которые вы используете
- Новые возможности, упрощающие код

### 3. Обновляйте изолированно

Создайте отдельную ветку. Обновляйте по одной мажорной зависимости за раз. Прогоняйте полный набор тестов. Так откат будет тривиальным, а отладка — целенаправленной.

```bash
git checkout -b chore/angular-21-upgrade
ng update @angular/core@21 @angular/cli@21
npm run test
npm run e2e
```

### 4. Сделайте обновления привычкой

Не ждите «подходящего момента» — его не будет. Выделите 2–4 часа ежемесячно именно на обновление зависимостей. Относитесь к этому как к любой другой технической задаче с дедлайном.

### 5. Поддерживайте тестовое покрытие

Обновления без тестов — это лотерея. Нужны:

- Unit-тесты для бизнес-логики
- Интеграционные тесты для критичных путей
- E2E-тесты для пользовательских сценариев

При низком покрытии его повышение окупается с каждым следующим обновлением.

## Советы по фреймворкам

### Angular

Используйте `ng update` — он запускает схемы, которые автоматически мигрируют код:

```bash
ng update @angular/core @angular/cli
```

Проверяйте [Angular Update Guide](https://angular.dev/update-guide) перед мажорными обновлениями. Там перечислены все breaking changes и шаги миграции.

### DevExtreme и UI-библиотеки

Они ломаются чаще, чем кажется. Фиксируйте точные версии в package.json при разработке. Проверяйте визуальные регрессии — автоматические скриншоты помогают.

### RxJS

Мажорные версии меняют операторы. При необходимости временно подключайте `rxjs-compat`, но удаляйте его сразу после миграции.

## Признаки, что вы слишком долго откладывали

- Отставание на несколько мажорных версий
- Несовместимость зависимостей зависимостей
- Уязвимости в дереве зависимостей
- Новые разработчики не находят актуальную документацию

## Скрытые преимущества

Регулярные обновления — это не только про избежание боли. Они дают:

- **Улучшение производительности** — фреймворки постоянно оптимизируют
- **Меньший размер бандла** — tree-shaking улучшается, мёртвый код удаляется
- **Лучший developer experience** — новые API обычно чище
- **Патчи безопасности** — уязвимости устраняются upstream

После обновления Angular 20→21 размер бандла уменьшился на 14%. Это бесплатная производительность для пользователей.

## Заключение

Обновление зависимостей похоже на спорт. Пропустить одну тренировку — нормально. Пропустить полгода — создать проблему, которую болезненно исправлять.

Лучшее время обновиться было вчера. Второе лучшее — сейчас, но маленькими регулярными шагами.

Начните на этой неделе. Выберите одну устаревшую зависимость. Обновите её. Повторите.
